# Product requirements

## Problem statement

Busy technical users want to stay aligned with important technology news without reading hundreds of RSS articles, release notes, and podcast descriptions. Existing RSS workflows are too noisy, so the user eventually stops following feeds and misses important headlines.

## Product vision

Wazzup provides an opinionated, personal briefing layer on top of RSS feeds and podcast releases. It continuously collects source material, selects items aligned with user interests, and publishes concise AI-assisted briefings at useful moments of the day.

## Goals

- Reduce time spent scanning feeds while preserving awareness of important technology news.
- Provide morning, hourly, and evening summaries that are short, source-linked, and aligned with configured interests.
- Run with minimal infrastructure to avoid maintaining servers or databases.
- Keep the frontend small, fast, installable, and easy to host statically.
- Standardize the data model and processing pipeline so future APIs, agents, and MCP integrations can reuse the same contracts.
- Include automated tests and GitHub Actions workflows from the start.

## Non-goals for the current product

- Building a general-purpose public news platform.
- Replacing full article reading or paid news subscriptions.
- Real-time breaking-news alerts with strict delivery guarantees.
- Multi-user SaaS account management.
- Training or fine-tuning an LLM.
- Hosting private user data on public GitHub Pages without an explicit privacy decision.

## Primary user

A technically proficient individual who follows technology news, open-source releases, security advisories, cloud/platform updates, and technical podcasts, but wants a curated summary instead of a raw feed reader.

## Assumptions validated by the implementation

- The first version serves one user or one small trusted group.
- GitHub Actions scheduling latency is acceptable for daytime collection every two hours.
- RSS and Atom are sufficient for the first implemented feed slice. JSON Feed, GitHub release feeds, and podcast RSS remain planned source types.
- Summaries can be generated by a pluggable AI runner such as Copilot CLI in GitHub Actions, Azure OpenAI, OpenAI, GitHub Models, Ollama, or another OpenAI-compatible endpoint.
- Copilot CLI is a viable GitHub-native automation path when authenticated with a GitHub account that has a valid Copilot license and a fine-grained token with Copilot Requests permission. In practice, the scheduled workflow must also be resilient when this secret is absent, so it falls back to the deterministic fake provider.

## Current implementation status

The current repository implements the end-to-end app slice with these concrete choices:

- Runtime: Python package under [../src/wazzup](../src/wazzup) with `PyYAML` as the only runtime dependency.
- Frontend: vanilla static PWA under [../public](../public).
- Automation: [../Taskfile.yml](../Taskfile.yml) as the workflow boundary, with [../.mise.toml](../.mise.toml) pinning Python, Task, and lint tools.
- Data: YAML is canonical in `public/data`; JSON mirrors are generated for browser consumption.
- State: `news-state` GitHub Release asset `wazzup-state.zip`, not generated commits.
- CI/CD: PR/manual CI, PR/manual reusable lint, scheduled/manual news generation, and Pages deploy after successful News hourly.

Implemented deviations from the original target:

- Automatic morning/evening due-time scheduling is implemented. The pipeline supports `--force-briefing auto|hourly|morning|evening`, and workflow dispatch exposes those options.
- JSON Feed and podcast source parsing are deferred.
- Direct Azure OpenAI/OpenAI/GitHub Models/Ollama providers are deferred; only `fake` and `copilot-cli` are implemented.
- AI budget environment variables are configured in workflow YAML but not enforced in code yet.
- Formal JSON Schema files are not present yet; runtime validators and tests enforce the current data contract.

## Current scope

### Included

- Static configuration for feeds and interests.
- Daytime two-hour RSS collection via GitHub Actions.
- Normalized article store as YAML files with generated JSON browser mirrors.
- AI provider abstraction with Copilot CLI and deterministic fake provider implementations.
- Morning and evening briefing generation when due in `auto` mode or explicitly forced.
- GitHub Pages-hosted frontend.
- Public GitHub Pages deployment is acceptable for the current deployment.
- CI workflow for lightweight formatting, syntax linting, tests, build checks, fixture generation, and data contract validation.
- End-to-end thin slice first: source config, ingestion, scoring, Copilot CLI summarization adapter, static data, PWA view, and CI.

### Deferred

- True multi-user accounts.
- Web Push at scale.
- Native mobile apps.
- Full-text article extraction for all sources.
- Automated podcast audio transcription unless transcript URLs already exist.
- MCP server process and public REST API implementation.
- Non-PWA delivery channels such as Home Assistant, ntfy, email, Teams, or Slack.
- Direct API AI providers and local Ollama/Foundry runners.
- Formal schema files, prompt regression fixtures, and provider token/monthly cost enforcement.

## Recommended answers to open questions

### How to reach the user?

Start with a PWA only for the current version. Later, expand to low-maintenance delivery channels:

1. GitHub Pages PWA for browsing and offline reading.
2. Optional GitHub Actions delivery step to send morning/evening summaries to ntfy, email, Slack, Teams, or Home Assistant webhook.
3. Browser notifications only when the installed PWA's browser supports background sync; unsupported platforms such as iOS should show a clear unavailable state.

Full Web Push requires subscription storage and a push sender. That introduces backend state, so it should be a later feature.

### Where to run the backend?

Use GitHub Actions for the current backend. Publish generated static data to GitHub Pages. Avoid a database initially by storing date-partitioned YAML files, generated JSON browser mirrors, and small indexes in a dedicated GitHub Release asset that the scheduled workflow restores before each run.

### Can the backend run on GitHub without maintaining a database?

Yes, with trade-offs. Recommended current storage:

- `data/latest.yaml` for the latest briefing pointers, with `latest.json` as browser mirror.
- `data/briefings/YYYY/MM/DD/*.yaml` for generated summaries, with JSON mirrors.
- `data/articles/YYYY/MM/DD.yaml` for normalized article records, with JSON mirrors.
- `data/sources/status.yaml` for feed health, with JSON mirror.
- `news-state` GitHub Release asset for the rolling 35-day state window.
- Optional monthly GitHub Release archives for cold history or monthly recaps.

This avoids an operational database and avoids thousands of generated-data commits per year. A separate `news` branch is possible, but the release asset is cleaner because it keeps generated state out of Git history.

### How to run the frontend with as few libraries as possible?

The implemented app uses static HTML, CSS, vanilla JavaScript, and a Service Worker. TypeScript, Web Components, or a framework can be added later only if they add clear value while keeping dependencies low, stable, and well-known.

### How to embed workflows from `.github` repositories?

Use reusable workflows for shared CI and release behavior. Keep project-specific schedule and secrets in this repository, and call organization-level workflows with `uses: owner/.github/.github/workflows/workflow-name.yml@version`.

### What Home Assistant integration is useful?

Start with a Home Assistant REST sensor or webhook consuming `latest.json`. Add optional MQTT discovery later. A morning automation can read the selected briefing through Home Assistant TTS, including Microsoft/Azure TTS if configured in Home Assistant.

### How to integrate backend with AI?

Use an explicit AI summary gateway interface in the pipeline. The default production adapter is Copilot CLI because it runs naturally inside GitHub Actions and supports non-interactive programmatic prompts. Keep Azure OpenAI, OpenAI, GitHub Models, Ollama, Foundry, and OpenAI-compatible endpoints as future adapter options. The pipeline should never depend directly on one provider SDK or CLI in business logic.

### Can GitHub Actions ask GitHub Copilot to summarize?

Yes. GitHub documents running Copilot CLI in GitHub Actions by installing `@github/copilot`, authenticating with `COPILOT_GITHUB_TOKEN`, and invoking `copilot -p` with `--no-ask-user` plus narrowly scoped `--allow-tool` options. For Wazzup, Copilot CLI is implemented as an AI runner adapter that receives a prepared prompt bundle and writes structured JSON for validation. The workflow and provider currently enforce runtime structure/citation checks, prompt-size item caps, and secret hygiene; formal JSON Schema files and token/monthly budget enforcement remain deferred.

### Should we consider Ollama in GitHub Actions?

Yes, but as a secondary/self-hosted style option. Ollama can run local models inside Actions for privacy-friendly experiments and deterministic smoke tests, but GitHub-hosted runners are CPU-bound, model downloads are slow without caching, and small local models may produce lower-quality summaries. Use it behind the same adapter interface, not as the default path.

### How to support podcasts?

Add podcast RSS as a source adapter. Prefer existing episode transcripts from Podcasting 2.0 transcript tags or linked show notes. Use audio transcription only as an opt-in expensive enhancement.

### How to standardize functionality?

Define versioned contracts for sources, content items, summaries, scores, and delivery targets. Build every interface around those contracts so the CLI, frontend, future API, agents, and MCP server share the same domain model.

## Resolved implementation decisions

| Decision             | Current choice                                                                                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Data visibility      | Public GitHub Pages is acceptable for the current deployment.                                                                                           |
| AI runner            | Copilot CLI is the default; explore Ollama, Foundry, or API providers later.                                                                            |
| Primary delivery     | PWA first.                                                                                                                                              |
| Frontend stack       | Implemented as vanilla HTML/CSS/JavaScript with no frontend build step. TypeScript/Web Components can be introduced later only if they add clear value. |
| Storage retention    | Keep 35 days of Pages data to support future monthly recaps with buffer.                                                                                |
| Summary language     | Always generate English briefings, including summaries of Dutch NOS articles.                                                                           |
| Implementation order | Build an end-to-end thin slice first.                                                                                                                   |
| Time zone            | Configured IANA time zone, default `Europe/Amsterdam`.                                                                                                  |
| Generated state      | Store generated data in the `news-state` release asset, not commits on `main` or a `news` branch.                                                       |
| Workflow automation  | Keep shell logic in Task and use reusable `.github` workflows where practical.                                                                          |

## Success metrics

- Morning briefing can be read in under 3 minutes.
- Evening briefing can be read in under 5 minutes.
- At least 90% of summaries include source citations.
- Feed collection succeeds for at least 95% of configured sources over 7 days.
- AI provider cost remains within the configured monthly budget once token/monthly accounting is implemented.
- The user reopens the briefing at least 4 days per week after the first month.
